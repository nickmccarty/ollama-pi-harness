# Experiment 01: Pipeline Generalization Study

## Objective

Determine whether the dual-search harness produces consistent quality across different task
types, count constraints, and topic domains. All previous runs used the same task (verification
loop patterns). This experiment tests generalization.

---

## Design

**Type:** One-factor completely randomized design (CRD)
**Factor:** Task type (3 levels: T_A, T_B, T_C)
**Replications:** 3 per task
**Total runs:** 9
**Run order:** Randomized to prevent systematic bias from search result drift or model cache effects

---

## Tasks

| ID  | Count constraint | Domain              | Output path                          |
|-----|-----------------|---------------------|--------------------------------------|
| T_A | Top 5 (explicit) | Context engineering | harness-engineering/context-engineering.md |
| T_B | Open-ended       | Cost management     | harness-engineering/cost-management.md     |
| T_C | Top 3 (explicit) | Agent failure modes | harness-engineering/agent-failure-modes.md |

**Full prompts:**

T_A: "Search for the top 5 context engineering techniques used in production LLM agents and save to ~/Desktop/harness-engineering/context-engineering.md"

T_B: "Search for best practices for cost envelope management in production AI agents and save to ~/Desktop/harness-engineering/cost-management.md"

T_C: "Search for the 3 most common failure modes in multi-agent AI systems and save to ~/Desktop/harness-engineering/agent-failure-modes.md"

## Response Variables

Collected automatically via `runs.jsonl` / `analytics.py`:

| Variable             | Description                                      | Direction |
|----------------------|--------------------------------------------------|-----------|
| `output_bytes`       | Size of final written file                       | higher = richer |
| `output_lines`       | Line count of final file                         | higher = more structured |
| `first_wiggum_score` | Evaluator score on round 1 (pre-revision)        | higher = better first pass |
| `wiggum_rounds`      | Number of evaluate/revise cycles needed          | lower = better |
| `final`              | PASS / FAIL / ERROR                              | PASS = success |
| `total_search_chars` | Merged chars from both searches                  | diagnostic |
| `quality_floor_hit`  | Whether fallback search was triggered            | diagnostic |

---

## Controlled Variables (held constant across all runs)

- Producer model: `pi-qwen` (qwen2.5:7b)
- Evaluator model: `glm4:9b`
- Wiggum max rounds: 3
- Wiggum pass threshold: score >= 8
- Searches per task: 2
- Search quality floor: 1800 chars
- Max search results per query: 5

---

## Hypotheses

**H1 (generalization):** Mean `output_bytes` will not differ significantly across task types.
The harness produces consistent quality regardless of domain or count constraint.

**H2 (count constraint effect):** Tasks with explicit count constraints (T_A, T_C) will have
lower `wiggum_rounds` variance than the open-ended task (T_B). An explicit count gives the
evaluator a concrete criterion to check.

**H3 (pass rate):** All 9 runs will achieve `final = PASS`. The wiggum loop should recover
from any first-pass failures.

**H4 (search floor):** `quality_floor_hit` will be rare (0-1 of 9 runs). Dual search
consistently produces > 1800 chars based on prior data.

---

## Analysis Plan

After all 9 runs complete:

1. **Per-task descriptive statistics:** mean and std dev of `output_bytes`, `output_lines`,
   `first_wiggum_score`, `wiggum_rounds` for each task.

2. **Coefficient of variation (CV):** `std / mean` per task for `output_bytes` — measures
   within-task consistency. CV < 20% = pipeline is stable for that task type.

3. **Count constraint comparison:** Compare `wiggum_rounds` distributions between
   constrained tasks (T_A, T_C) and unconstrained (T_B).

4. **Pass rate:** Count PASS vs FAIL vs ERROR across all 9 runs.

5. **Search quality vs output quality:** Correlate `total_search_chars` with `output_bytes`
   across runs to test whether the dual-search hypothesis holds at larger sample size.

---

## What Would Falsify Each Hypothesis

- H1 falsified: output_bytes CV > 40% across task types, or one task type consistently
  scores < 1000 bytes while others score > 2000.
- H2 falsified: T_B has the same or lower wiggum_rounds variance as T_A and T_C.
- H3 falsified: any run reaches `final = FAIL` (max wiggum rounds without passing).
- H4 falsified: quality floor is hit on more than 2 of 9 runs.

## Findings

### H1 (generalization) - partially falsified

T_B (open-ended, cost management) was the most stable task: CV = 13.2%, well below the 20% stability threshold. T_A (top 5) was borderline at 39.7%, approaching but not exceeding the 40% falsification threshold. T_C (top 3) was unstable at CV = 44.2%, exceeding the threshold.

The instability in T_C (1291 to 2649 bytes across three runs) is notable because it is the most constrained task — "top 3" should anchor the output structure. Instead, the model interpreted the count constraint loosely in Run 7, producing 7 failure modes despite the explicit "3" in the prompt. That run (2649 bytes) drove the variance. The evaluator (glm4:9b) passed it at 9/10 despite the count violation — the count constraint rule in the wiggum prompt was not enforced.

Counterintuitive finding: the open-ended task (T_B) was the most consistent. An unconstrained task appears to give the model more latitude to produce proportionally structured output, while explicit count constraints introduce brittleness when the model over-delivers.

### H2 (count constraint effect) - indeterminate

All 9 runs produced `wiggum_rounds = 1` with `score_r1 = 9`. The metric is degenerate: zero variance across all tasks and all runs. H2 cannot be confirmed or falsified — the wiggum loop never needed to revise, so there is no signal on which to compare constrained vs unconstrained tasks.

The ceiling effect on wiggum scores (9/10 on every first pass) suggests either the evaluator criteria are now slightly too lenient at the margin, or the dual-search harness has raised baseline output quality to the point where revision is rarely needed. A stronger evaluator or a higher pass threshold (e.g., 9 instead of 8) would be required to generate meaningful variance in this metric.

### H3 (pass rate) - confirmed

9/9 PASS. No run reached `final = FAIL`. H3 confirmed unconditionally.

### H4 (search floor) - confirmed

0 of 9 runs triggered the quality floor fallback. Merged dual-search results ranged from 2952 to 3577 chars — all comfortably above the 1800-char floor. H4 confirmed.

### Search quality vs output quality correlation

Pearson r = -0.577 between `total_search_chars` and `output_bytes` across the 9 experiment runs — a moderate negative correlation, contrary to the positive relationship observed in the prior 6-run study.

Interpretation: the prior study compared single-search (1414-2326 chars) against dual-search (3198-3424 chars), where the regime shift drove both variables. In this experiment, search chars are compressed into a narrow band (2952-3577 chars, range = 625 chars) with no single-search runs. Within that narrow range, search volume is no longer a meaningful predictor. The negative sign is noise, not a real effect. The prior finding ("search chars is a leading indicator of output quality") holds at regime scale but not within a tight range where the floor has already been cleared.

### Anomaly: count constraint not enforced by evaluator

Run 7 (T_C, "top 3 failure modes") produced an output containing 7 distinct failure modes. The wiggum evaluator scored it 9/10 and passed it, despite the count constraint rule in the scoring prompt ("if the output contains the wrong number of items, cap the score at 5 and set passed=false"). This represents an evaluator reliability gap: glm4:9b does not consistently apply structured rules when the output otherwise reads as high quality. A stronger evaluator or a pre-wiggum harness-side count check would be needed to enforce count constraints reliably.
